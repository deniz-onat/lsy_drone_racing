  ---
  STEP 1: Understanding the controller

  1. What does it do at a high level?
  The controller flies the drone through 4 gates in a fixed order. It draws a
  smooth curved path (a cubic spline — think of it as a smooth ribbon) through
  all the gates from wherever the drone currently is, then tracks that ribbon by
   sending attitude commands (tilt left, tilt right, thrust up/down) 50 times
  per second.

  2. How does it plan a path?
  It uses a cubic spline — a mathematical curve that passes smoothly through a
  sequence of 3D waypoints. For each gate it computes three waypoints: one
  before the gate (approach), the gate center itself, and one after (exit). It
  inserts extra "clearance" waypoints between gates that are at different
  heights or require sharp turns. Obstacles are avoided by nudging waypoints
  sideways until they clear the obstacle by a safe margin.

  3. What information does it receive every timestep?
  - pos — where the drone is (x, y, z)
  - vel — how fast it's going (x, y, z)
  - quat — which direction it's pointing (as a quaternion)
  - target_gate — which gate it's currently trying to fly through (0, 1, 2, 3,
  or -1 when done)
  - gates_pos — the position of every gate (4 × 3D point)
  - gates_quat — the facing direction of every gate
  - gates_visited — a True/False array: has each gate ever been within sensor
  range?
  - obstacles_pos — where the 4 obstacles are
  - obstacles_visited — same True/False logic as gates

  4. How does it use gate and obstacle positions?
  Gate positions are used as the "must-pass-through" points for the spline.
  Obstacle positions are used to push waypoints sideways so the spline stays 20
  cm away from every obstacle in the horizontal plane. Every time the controller
   detects that a position has shifted by more than 5 cm from what it last
  planned with, it rebuilds the entire spline from scratch from the current
  drone position.

  5. What does gates_visited mean, and how is it used?
  Each entry in gates_visited is False until the drone flies within 0.7 m
  (horizontal distance only) of that gate — then it permanently flips to True.
  While a gate is False, the observation reports the gate's nominal
  (pre-configured) position. Once True, it reports the actual position. The
  controller itself doesn't read gates_visited directly — it only notices that
  positions changed (when the flag flips, the reported position jumps from
  nominal to real) and that triggers an automatic replan.

  6. What is the sensor range, and how does the controller react?
  Sensor range is 0.7 m horizontal distance (not 3D — altitude doesn't matter
  for triggering it). When any gate or obstacle enters that range, the
  observation immediately starts reporting its true position. Since the true
  position differs from the nominal by up to ~15 cm for Level 2, the
  controller's position-change check sees the shift and rebuilds the spline with
   the corrected data.

  7. How does it update the path?
  ReferenceManager._needs_plan() runs every timestep and checks: did the target
  gate change? Did any gate position shift more than 5 cm? Did any obstacle
  position shift more than 5 cm? If any answer is yes, it rebuilds the full
  spline from the drone's current position through all remaining gates.

  8. What does it assume at the start — does it already know gate positions?
  Yes. The very first observation provides gates_pos for all 4 gates. On Level
  2, these are the nominal positions from the config file (which are close to
  the real positions). So on the very first tick, the controller builds a full
  spline through all 4 gates and starts flying. It never needs to "discover"
  them — it assumes it basically knows where they are.

  9. After the last gate, what does the drone do?
  When all gates are passed, the environment sets target_gate = -1. The
  controller sees this, sets _finished = True, and just returns its last
  computed attitude command — which happens to be whatever it was doing when it
  exited gate 3. In practice this means the drone coasts forward and then
  crashes or hovers briefly. There is no explicit return-to-home logic.

  ---
  STEP 2: Why the current controller fails on Level 3

  The core problem: Level 3 puts all four gate nominal positions at exactly [0,
  0, height] — the origin. Since the controller immediately plans a spline
  through all 4 gates on tick 1, it tries to build a path through four points
  all stacked at the same location. The spline planner either crashes, produces
  a degenerate curve, or sends the drone on a suicidal dive into the origin.

  Specific code that assumes known nominal positions:

  - parse_observation() in state.py:41-61 — reads gates_pos directly from the
  observation without checking whether any gate has been visited yet. If it's at
   origin, that's what gets used.
  - ReferenceManager.build() in trajectory.py:232-258 — immediately calls
  build_waypoints() with all gate positions, including fake origin positions for
   unvisited gates.
  - build_waypoints() in trajectory.py:135-208 — loops over
  gates_pos[target_gate:] assuming all entries are meaningful real-world
  coordinates.
  - KaFa_1500_v6.__init__ doesn't record a home position, and compute_control in
   KaFa_1500_v6.py:65 calls ensure_plan() on tick 1 — before the drone has
  discovered anything.
